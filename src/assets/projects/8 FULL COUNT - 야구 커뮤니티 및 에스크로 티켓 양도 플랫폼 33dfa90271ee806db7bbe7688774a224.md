## ***Overview***

---

FULL COUNT는 같은 팀을 응원하는 팬들이 경기 일정에 맞춰 직관 메이트와 크루를 모집하고, 실시간 채팅으로 소통하며, 개인 간 티켓 양도를 에스크로 형식으로 안전하게 진행할 수 있도록 돕는 서비스입니다. KBO 경기 일정과 경기 현황을 기반으로 오프라인 직관 경험을 온라인 커뮤니티와 연결합니다.

## *Problem Solving*

---

### 1. 채팅방 참여 구조 최적화 및 무결성 확보

- **문제**: 채팅방(N)과 회원(M) 간의 다대다 관계에서 발생하는 데이터 복잡성 및 특정 회원이 동일 채팅방에 중복 입장하는 데이터 결함 가능성.
- **해결**:
    - **연결 엔티티(Mapping Entity) 도입**: `ChatRoomParticipant`를 통해 다대다 관계를 1:N, N:1로 해소하고 도메인 모델의 유연성 확보.
    - **복합 유니크 제약 조건**: 데이터베이스 레벨에서 `(chat_room_id, member_id)` 복합 인덱스를 설정하여 물리적인 중복 참여를 근본적으로 차단.
    - **지연 로딩(LAZY) 전략**: 모든 연관 관계에 지연 로딩을 기본 적용하여 메시지 송수신 시 불필요한 회원/채팅방 정보가 로딩되는 오버헤드 방지.

### 2. JPA N+1 문제 해결 및 페치 조인(Fetch Join) 최적화

- **문제**: 채팅 목록이나 메시지 조회 시 연관된 엔티티(작성자, 게시글 등)를 개별적으로 조회하면서 발생하는 과도한 Select 쿼리(N+1) 현상.
- **해결**:
    - **선택적 페치 조인**: 기본 전략은 LAZY를 유지하되, 성능이 중요한 조회 API에서만 `JOIN FETCH`를 사용하여 단 한 번의 쿼리로 연관 데이터를 일괄 수집.
    - **용도별 조회 쿼리 분리**: 전체 데이터를 가져오는 `Details` 쿼리와 핵심 정보만 가져오는 `Summary` 쿼리를 분리 설계하여 메모리 및 네트워크 리소스 낭비 최소화.

### 3. 대용량 메시지 조회를 위한 커서 기반 페이징

- **문제**: 채팅 메시지 데이터 누적 시 기존 Offset 기반 페이징은 뒤로 갈수록 조회 속도가 느려지며, 데이터 추가/삭제 시 목록이 밀리는 현상 발생.
- **해결**:
    - **No-Offset 커서 조회**: 메시지 ID를 기준으로 하는 커서 기반 조회를 도입하여 인덱스 스캔 효율을 극대화하고 대용량 데이터에서도 일정한 응답 속도 유지.
    - **Slice 인터페이스 활용**: 전체 데이터 개수를 계산하는 불필요한 `COUNT` 쿼리를 생략하고, 다음 페이지 존재 여부만 확인하여 데이터베이스 부하 경감.

### 4. 비즈니스 로직 효율화 및 DB 라운드 트립 최소화

- **문제**: 채팅방 목록 화면 구성 시 각 방의 마지막 메시지를 개별적으로 조회하거나, 권한 확인을 위해 전체 객체를 로딩하는 비효율적인 통신 구조.
- **해결**:
    - **마지막 메시지 일괄 조회**: 서브쿼리와 `IN` 절을 조합하여 여러 채팅방의 최신 메시지를 단일 쿼리로 조회함으로써 DB 연결 횟수(Round-trip)를 획기적으로 감소.
    - **경량 존재 확인(Exists) 쿼리**: 전체 엔티티를 영속성 컨텍스트에 올리지 않고 `EXISTS` 쿼리를 통해 참여 여부만 `boolean`으로 즉시 판단하여 검증 로직 속도 향상.

### 5. 쿼리 실행 가시성 확보 및 배치 조회를 통한 N+1 완화

- **문제**: JPA/Hibernate 사용 시 실제 발행되는 SQL의 동작 방식을 파악하기 어렵고, 연관 엔티티 로딩 시 발생하는 지연 로딩으로 인해 성능 병목(N+1)이 불규칙하게 발생함.
- **해결**:
    - **로그 포맷팅 및 SQL 출력**: `show-sql`과 `format_sql` 설정을 통해 쿼리 구조를 직관적으로 분석하고, 비효율적인 조인이나 조건문을 개발 단계에서 즉시 파악.
    - **Batch Fetch 설정**: `default_batch_fetch_size: 100`을 적용하여 연관된 자식 엔티티 조회 시 `IN` 절을 통해 묶음 조회함으로써, 100번 발생할 쿼리를 1번으로 획기적으로 줄여 N+1 문제의 범용적 해결책 마련.

### 6. 환경별 프로필 분리 및 데이터베이스 최적화

- **문제**: 개발과 운영 환경이 동일한 설정을 공유할 경우, 보안 취약성(계정 노출) 및 개발 편의성 저하(무거운 DB 연결) 문제가 발생함.
- **해결**:
    - **Multi-Profile 전략**: `dev`(H2 인메모리)와 `prod`(MySQL) 프로필을 분리하여 개발 시에는 빠른 피드백 루프를, 운영 시에는 데이터 안정성을 확보.
    - **보안 및 초기화 자동화**: 운영 환경의 민감 정보는 환경 변수로 관리하고, 개발 환경에서는 `data.sql` 및 H2 콘솔을 활성화하여 테스트 데이터 적재 및 검증 효율 극대화.

### 7. 병목 구간 진단 및 실시간 모니터링 체계

- **문제**: 목록 조회, 통계 산출 등 복잡한 비즈니스 로직에서 불필요한 반복 조회나 카운트(Count) 쿼리의 오버헤드가 누적되어 전체적인 응답 속도가 저하됨.
- **해결**:
    - **로그 기반 병목 진단**: 특정 패키지(`AdminService` 등)의 로그 레벨을 조정하여 비즈니스 로직 수행 시 발생하는 SQL의 수와 실행 시간을 정밀 모니터링.
    - **쿼리 최적화 가이드라인**: 모니터링 결과를 바탕으로 페이징 처리 시 불필요한 전체 카운트 쿼리를 방지하고, 중복 조회가 발생하는 구간에 DTO 직접 조회나 페치 조인(Fetch Join)을 선택적으로 적용하여 시스템 신뢰도 향상.
