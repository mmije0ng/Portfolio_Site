## *Problem Solving*

---

### 1. 소비 루틴 동시성 이슈 해결

- **문제**: 예산당 월 1개의 소비 루틴만 허용되는 구조에서, 동일 월에 대한 동시 등록 요청 시 중복 생성 가능성 발생
- **해결**:
    - `(user_id, budget_id, created_month)` 조합에 **DB 유니크 제약** 추가하여 데이터 무결성 보장
    - 루틴 등록 시 대상 범위에 **PESSIMISTIC_WRITE(비관적 락)**을 적용하여 레이스 컨디션 제거
    - 애플리케이션 레벨의 락과 DB 레벨의 제약 조건을 결합한 **이중 안전장치** 구성
- **결과**: QA 환경에서 중복 루틴 생성 문제를 완벽히 방지하여 데이터 정합성 확보

### 1-1) 비관적 락 VS 낙관적 락

소비 루틴 등록 시 **낙관적 락** 대신 **비관적 락**을 선택한 이유는 크게 세 가지 전략적 판단 때문입니다.

### **1. 중복 생성(Insert) 방지의 한계 극복**

- **낙관적 락의 한계**: 주로 `@Version`을 이용한 **수정(Update)** 시점에 충돌을 감지합니다. 하지만 '루틴 등록'은 데이터가 없는 상태에서 새로 생성하는 것이므로, 버전 관리만으로는 찰나의 순간에 동시에 들어오는 두 개의 `Insert` 요청을 완벽히 제어하기 어렵습니다.
- **비관적 락의 이점**: `SELECT ... FOR UPDATE`를 통해 조회 단계부터 해당 범위에 잠금을 겁니다. 데이터가 생성되기 전부터 통로를 좁혀 **레이스 컨디션(Race Condition)**을 원천 차단합니다.

### **2. 사용자 경험(UX)과 재시도 비용 절감**

- **낙관적 락**: 충돌 발생 시 예외를 던지며 트랜잭션을 종료합니다. 이를 처리하려면 개발자가 별도의 **재시도(Retry) 로직**을 구현해야 하며, 사용자 입장에서는 처리가 취소되는 경험을 할 수 있습니다.
- **비관적 락**: 먼저 온 요청이 끝날 때까지 두 번째 요청이 **DB 레벨에서 대기**합니다. 결과적으로 별도 로직 없이도 순차 처리가 보장되어 시스템 안정성이 높아집니다.

### **3. 잠금 범위의 최소화로 성능 효율 확보**

- **비관적 락의 단점**: 흔히 성능 저하가 우려된다고 하지만, 본 프로젝트에서는 락의 범위를 `(user_id, budget_id, month)`로 매우 좁게 설정했습니다.
- **판단**: 특정 사용자의 특정 예산에 대해서만 잠금이 발생하므로, 전체 시스템 성능에 미치는 영향은 미비하면서도 **데이터 무결성**이라는 가계부 서비스의 핵심 가치를 지킬 수 있다고 판단했습니다.

| **비교 항목** | **낙관적 락 (Optimistic)** | **비관적 락 (Pessimistic)** |
| --- | --- | --- |
| **작동 원리** | 충돌이 안 날 것이라 가정 (버전 체크) | 충돌이 날 것이라 가정 (선점 잠금) |
| **제어 시점** | 트랜잭션 **커밋** 시점 | 트랜잭션 **조회** 시점 |
| **실패 처리** | 예외 발생 시 직접 롤백/재시도 구현 | 선행 트랜잭션 완료까지 자동 대기 |
| **선택 이유** | - | **데이터 정합성 최우선 & 순차 처리 보장** |

### 1-2) `(user_id, budget_id, created_month)` 조합에 **DB 유니크 제약** 추가하여 데이터 무결성 보장

```java
@Getter
@Builder
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@AllArgsConstructor
@Entity
@Table(
        uniqueConstraints = {
                @UniqueConstraint(
                        name = "unique_user_budget_month",
                        columnNames = {"user_id", "budget_id", "created_month"}
                )
        }
)
public class Routine extends BaseEntity {
		// 생략

    // 생성일, "YYYY-MM" 형식으로 저장 (예: 2025-08)
    @Column(name = "created_month", nullable = false, length = 7)
    private String createdMonth;

    // 소비루틴-예산 다대일
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "budget_id")
    private Budget budget;

    // 소비루틴-유저 다대일
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    private User user;
}
```

```java
-- 유니크 제약(= 유니크 인덱스)
ALTER TABLE routine
  ADD CONSTRAINT unique_user_budget_month
  UNIQUE (user_id, budget_id, created_month);
```

- `(user_id, budget_id, created_month)` 조합이 **중복 저장**되면 DB단에서 막기
- 애플리케이션 레벨 검사 사이에 경쟁이 있어도, **최종적으로 DB가 중복 Insert를 거부**
- 운영에서는 이 제약 위반을 `DataIntegrityViolationException` 등으로 감싸 사용자에게 “이미 이번 달 루틴 존재” 오류를 리턴

### 1-3) 루틴 등록 시 대상 범위에 **PESSIMISTIC_WRITE(비관적 락)**을 적용하여 레이스 컨디션 제거

```java
    // 소비 루틴 등록
    @Transactional
    @Override
    public RoutineResponse.RoutineCreateResultDTO saveRoutineWithRoutineHashtags(
            Long userId,
            Long budgetId,
            RoutineRequest.RoutineCreateDTO request) {
        // 1. 사용자 조회
        User user = userRepository.findById(userId)
                .orElseThrow(() -> new ErrorHandler(ErrorStatus.USER_NOT_FOUND));

        // 2. 예산 조회
        Budget budget = budgetRepository.findByIdAndUserId(budgetId, userId)
                .orElseThrow(() -> new ErrorHandler(ErrorStatus.BUDGET_NOT_FOUND));

        // 3. 예산의 createdMonth로 이번 달 판단
        String createdMonth = budget.getCreatedMonth();

        // 4. 이번 달 동일 예산에 대한 루틴 존재 여부 확인 (PESSIMISTIC_WRITE)
        if (routineRepository.findForUpdateByUserAndBudgetAndMonth(userId, budgetId, createdMonth).isPresent()) {
            throw new ErrorHandler(ROUTINE_ALREADY_EXIST);
        }
        
        // 생략
        
        // 7. 루틴 저장
        //   - 동시에 같은 (user, budget, createdMonth) 조합이 저장되는 경쟁 상황 방지
        //   - 유니크 제약 위반 발생 시 DataIntegrityViolationException을 잡아 예외 변환
        Routine routine = RoutineConverter.toRoutine(user, budget, request, createdMonth);
        try {
            routineRepository.save(routine);
        } catch (DataIntegrityViolationException e) {
            // 혹시 경쟁 상황으로 유니크 제약 위반 시
            throw new ErrorHandler(ROUTINE_ALREADY_EXIST);
        }
        
     }
```

```java
public interface RoutineRepository extends JpaRepository<Routine, Long>, RoutineRepositoryCustom {
    // 현재 예산(budgetId)과 연월(createdMonth)에 해당하는 루틴이 존재하는지 확인 + PESSIMISTIC_WRITE 락
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT r FROM Routine r " +
            "WHERE r.user.id = :userId " +
            "AND r.budget.id = :budgetId " +
            "AND r.createdMonth = :createdMonth")
    Optional<Routine> findForUpdateByUserAndBudgetAndMonth(
            @Param("userId") Long userId,
            @Param("budgetId") Long budgetId,
            @Param("createdMonth") String createdMonth
    );
}
```

- **쓰기 트랜잭션 시작 시, 후보 레코드에 잠금(PESSIMISTIC_WRITE)을 건다.**
- 존재하면 바로 “이미 있음”으로 실패.
- 존재하지 않으면 **같은 조건으로 들어오는 다른 트랜잭션을 대기시키고** 내가 먼저 INSERT.
- 혹시 동시에 INSERT를 시도해도 **DB 유니크 제약**이 최종 방어.
- 루틴 등록 트랜잭션에서 **해당 (user, budget, month)** 범위를 `SELECT ... FOR UPDATE`로 **잠금**.
- 동시에 같은 조건으로 등록을 시도한 트랜잭션은 **락 대기**에 들어가고, 먼저 완료된 트랜잭션의 결과를 보고 뒤에 온 트랜잭션은 중복으로 판단되어 실패하게 됨
- 즉, **중복 체크와 insert 사이의 레이스 컨디션을 제거**

### 2. Bastion Host 기반 DB 접근 제어

![image.png](image%2059.png)

- **문제:** 보안상 프라이빗 서브넷에 배치한 RDS에 개발자들이 직접 접근이 불가능해진 운영상 문제 발생
- **해결**:
    - 퍼블릭 서브넷에 **Bastion Host를 구성**하고 SSH 터널링/포트 포워딩 방식으로 접근 경로 설계
    - 인증된 사용자만 Bastion Host를 경유하도록 하여 DB 보안 구조 강화
- **결과**: 외부 노출 없이 안전한 DB 접근 환경을 구축하여 보안성과 운영 안정성을 동시에 확보
